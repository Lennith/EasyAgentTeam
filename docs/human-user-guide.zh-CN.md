# 给普通用户的使用说明

这套系统默认是给 Agent 来部署、运行和协作的。

如果你不会写代码，也没有关系。你的主要工作不是自己操作很多命令，而是把目标讲清楚，让 Agent 帮你完成。

## 你可以把它理解成什么

EasyAgentTeam 可以理解成一个“多位 AI 助手一起分工干活”的系统。

它通常会做这几件事：

- 帮你启动项目
- 帮你安排多个 Agent 分工
- 帮你执行任务或流程
- 帮你记录过程，方便回头看

你不需要先理解全部技术细节。先记住一句话：

`你负责提需求，Agent 负责部署、运行、检查结果。`

## 最常见的使用方式

### 0. 你可以这样使用它（批量邮箱处理工作流）

如果你想做一个由 5 个 Agent 同时来处理200封邮件并且生成回复草稿的工作流，你可以直接这样跟它说：

```text
我们来设计一个新的workflow，邮件工作流。我们已经导入了邮件skill，配置文件我们使用 config.json 这个。

首先是团队创建，团队有一个负责收取和建立联系人信息。一个负责对邮箱根据邮件内容进行分类，并且根据内容指定不同的Agent进行邮件内容处理和回复。3个不同的agent负责不同类型的邮件回复，一个负责产品信息类的信息回复。一个负责技术咨询相关的信息回复。一个负责其他邮件的回复。

回复邮件的原则是，只回复收件人 To 里包含我的邮箱的（不包含抄送），每次邮件处理都需要在团队工作区内留下一个统一格式的处理记录。设计需要防止第二次workflow执行时重复处理已经处理过的邮件。工作流程大概是收邮件-》分类指派回复-》设计回复，回复内容只落盘，不实际回发。预计回复放在outbox中，后面我们会用workflow外的agent手动来看要发的邮件和inbox报告，它不包含在workflow内。我们的计划需要完成团队设计，workflow流程设计，prompt设计。

另外回复邮件我们设计为并行处理，一个负责技术类邮件，一个负责产品类邮件，一个负责其他邮件。
```

你会得到什么：

- 一个可直接复用的 5-agent 邮件协作模板（handler / classifier / 3个reply）
- 一条可执行主流程：收件 -> 分类 -> 三路并行回复草稿 -> 统一记录
- 回复只落盘到 outbox，不会自动发信
- 团队共享目录统一（inbox/outbox/contacts/workflow），个人目录只留个人证据和进度
- 防重可用：跨 run 不重复处理、连续对话线程只保留一份主回复
- 可跑到 200 封规模（收件上限 200，并发 10）

### 1. 让 Agent 帮你部署

你可以直接对 Agent 说：

```text
请帮我把这个项目启动起来，并告诉我现在是否已经可以使用。
```

如果你想更明确一点，可以这样说：

```text
请帮我检查环境、安装依赖、启动项目，并告诉我下一步该看哪里。
```

### 2. 让 Agent 帮你介绍项目功能

如果你刚接触这个项目，可以这样说：

```text
请用普通人能听懂的话介绍这个项目能做什么，重点说我能怎么用。
```

或者：

```text
请带我快速了解这个项目的主要功能，不要讲太多代码细节。
```

### 3. 让 Agent 帮你运行一次标准验证

如果你想确认系统是不是正常，可以这样说：

```text
请帮我跑一轮标准检查，并告诉我结果是成功还是失败。
```

## 运行后你主要看什么

你通常只需要看这几类结果：

- `是否启动成功`
- `是否有报错`
- `任务是否完成`
- `系统有没有留下结果记录`

如果 Agent 说已经运行成功，你可以继续让它告诉你：

```text
请告诉我现在去哪里看运行结果，用最简单的方式说明。
```

## 如果你想看界面，重点看哪里

这个系统有很多页面，但普通用户先看下面几个就够了：

- `Projects`
  看项目任务有没有在推进
- `Workflow`
  看流程有没有在运行
- `task-tree`
  看任务结构和完成状态
- `timeline`
  看系统做过什么事
- `Settings`
  看配置是否大体正常

如果你不知道该看哪个页面，可以直接问 Agent：

```text
我现在应该看哪个页面来确认系统是否正常？
```

## 遇到问题时，最简单的说法

你不需要先判断技术原因，直接把现象告诉 Agent 就可以。

下面这些说法都很好用：

```text
项目没有启动起来，帮我看看卡在哪里。
```

```text
流程没有继续往下跑，帮我检查现在阻塞在什么地方。
```

```text
页面能打开，但我不知道结果算不算正常，帮我判断一下。
```

```text
请不要改太多东西，先观察和分析问题原因。
```

## 最常见的小问题

### 启动不了

可能是：

- 环境没准备好
- 依赖没装好
- 某个配置缺失
- 端口被占用

你可以直接说：

```text
请帮我检查为什么启动不了，优先告诉我最可能的原因。
```

### 流程没动

可能是：

- 某个任务在等待
- 某个 Agent 没有正常工作
- 某个配置不完整

你可以直接说：

```text
请帮我看为什么流程没有继续执行，先分析，不要急着大改。
```

### 看不懂结果

你可以直接说：

```text
请把这次运行结果翻译成普通人能理解的话，告诉我成功了没有。
```

## 推荐的提问方式

和 Agent 沟通时，尽量说清楚下面三点：

- 你想做什么
- 你现在看到什么现象
- 你希望 Agent 先做什么

一个好例子：

```text
我想确认这个项目能不能正常运行。现在页面已经打开，但我不知道系统是不是已经准备好。请先帮我检查运行状态，再告诉我下一步看哪里。
```

## 你不需要做的事

普通用户一般不需要自己做这些事：

- 不需要自己读大量代码
- 不需要自己判断复杂日志
- 不需要自己设计命令
- 不需要先懂项目架构再开始用

这些事情可以先交给 Agent。

## 一句话总结

把目标讲清楚，把看到的问题讲清楚，然后让 Agent 帮你部署、运行、检查和解释结果。
