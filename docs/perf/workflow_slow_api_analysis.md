# Workflow Slow API Analysis

## Scope

This note captures the current server-side perf trace analysis for workflow slow APIs.

- Trace switch: `WORKFLOW_PERF_TRACE=1`
- Raw trace path example:
  - `data/workflows/runs/e2e_gesture_run_20260414011913/audit/perf_trace.jsonl`
- Sample source:
  - Mixed-provider workflow E2E passing run `e2e_gesture_run_20260414011913`
  - This run finished with `workflow_runtime_ok` and `official_telemetry_pass=True`, so the measurements below reflect the current mixed baseline rather than an early broken sample.
- Server-side tracing currently emits only the raw `perf_trace.jsonl`.
- Aggregated artifacts such as `workflow_perf_summary.json` and `workflow_perf_report.md` are generated by `E2ETest/scripts/run-workflow-e2e.ps1`, not by the backend route handlers themselves.

### High-frequency spans

Trace counts from the passing run show the main read-path amplification:

- `workflowRuns.ensureRuntime`: 11948 calls
- `sessions.listSessions`: 5822 calls
- `workflowRuns.readRuntime`: 1559 calls
- `GET /task-runtime`: 183 calls
- `GET /task-tree-runtime`: 167 calls
- `dispatchRun`: 172 calls

This confirms the client-side `slow_api` warnings were not false positives. The server is repeatedly doing run/session/runtime work under polling and redispatch pressure.

## Interpretation Rules

- Route spans and nested service/repo spans are intentionally recorded together. Do not sum them directly.
- `route` spans represent the traced portion of the handler for covered workflow hotspots. For `POST /api/workflow-runs`, the span starts after `runId` is known, so template parsing and early validation are not fully covered.
- `service`, `repo`, and `view` spans explain where route time is spent.
- Because the trace is enabled globally by environment variable, high-frequency routes such as `task-runtime` and `task-tree-runtime` will naturally produce large trace volume.
- The current schema does not yet include `trace_id` / `parent_span_id`, so route-to-inner attribution is a hotspot correlation view rather than a strict per-request call tree.

## Hotspot Attribution

### 1. Polling routes re-read run/runtime/session state repeatedly

`GET /task-runtime` and `GET /task-tree-runtime` both do the same hot-path sequence:

1. `loadRunOrThrow(runId)` in `server/src/services/orchestrator/workflow/workflow-runtime-support-service.ts`
2. `getRun(runId)` -> `readRunRegistry(dataRoot)` -> full `runs.json` load in `server/src/data/repository/workflow/run-repository.ts`
3. `hydrateRunRuntime(dataRoot, run)` -> `readWorkflowRunTaskRuntimeState` in `server/src/data/repository/workflow/run-repository.ts`
4. `ensureRuntime(run)` in `server/src/services/orchestrator/workflow/workflow-runtime-support-service.ts`
5. `buildWorkflowRuntimeSnapshot` or `buildWorkflowTaskTreeView` in `server/src/services/orchestrator/workflow/workflow-run-query-service.ts`

This means every poll request reads run metadata, runtime state, and often converges runtime again even when nothing changed.

### 2. Session reads are cheap individually but too frequent

`sessions.listSessions` is only about 2.9 ms on average, but it appears 3077 times in the sampled run. The repository implementation always reopens `sessions.json` and reads the whole file under the run write lock in `server/src/data/repository/workflow/runtime-repository.ts`.

The problem is not one slow call. The problem is aggregate churn:

- repeated polling
- repeated dispatch/tick logic
- repeated session resolution inside the orchestrator

### 3. `patchRun` is a real write hotspot

`workflowRuns.patchRun` averages about 762 ms and peaks near 2 seconds. The implementation updates the entire run registry file and then re-hydrates runtime in `server/src/data/repository/workflow/run-repository.ts`.

This is expensive because:

- `runs.json` is a shared registry
- every patch writes the full registry
- `patchRun` is used on hot paths such as dispatch/state updates

### 4. `dispatchRun` cost is dominated by orchestration-side state work, not only provider latency

`dispatchRun` averages about 3.3 seconds with a max over 5.4 seconds. This is not explained by provider calls alone.

The trace pattern shows `dispatchRun` is surrounded by:

- repeated `ensureRuntime`
- repeated `patchRun`
- repeated `listSessions`
- repeated route polling

So the main cost is orchestration state preparation and persistence around dispatch, not a single provider bottleneck.

### 5. `ensureRuntime` is logically cheap, but the current call shape amplifies it

Repo-level `workflowRuns.ensureRuntime` is only around 4.8 ms on average, but it is called 6240 times. Service-level `ensureRuntime` can spike much higher because it may:

- read runtime
- converge task tree
- re-enter a workflow transaction
- write runtime back
- patch run metadata

Relevant logic is in `server/src/services/orchestrator/workflow/workflow-runtime-support-service.ts`.

The issue is call frequency plus occasional write-through behavior, not only raw function latency.

## Root Causes

1. Whole-file storage is on the hot path.
   - `runs.json`
   - `tasks.json`
   - `sessions.json`

2. Read APIs rebuild state too often.
   - load run
   - ensure runtime
   - rebuild snapshot/tree

3. Write APIs update broad state too often.
   - `patchRun`
   - `touchSession`
   - runtime convergence writes

4. Polling multiplies all of the above.
   - `task-runtime`
   - `task-tree-runtime`
   - session and dispatch observation loops

## Recommended Fix Plan

### Phase 1: Reduce duplicate read-path IO

Target: lower `task-runtime` and `task-tree-runtime` latency without changing storage format yet.

- Add a shallow run loader for hot read routes so they do not rehydrate full runtime twice.
- Reuse one loaded runtime per request instead of separate load/converge/rebuild cycles.
- Avoid redundant `ensureRuntime` calls inside the same request path.
- Add route-level memoization for current request scope when `runId` is the same.

Expected impact:

- reduce `GET /task-runtime`
- reduce `GET /task-tree-runtime`
- reduce `workflowRuns.readRuntime`
- reduce `workflowRuns.ensureRuntime`

### Phase 2: Reduce write amplification

Target: lower `dispatchRun`, `patchRun`, and session update overhead.

- Move hot session mutations away from rewriting the whole run registry.
- Keep runtime-specific state in run-local files only; avoid `patchRun` for metadata that already belongs to runtime/session files.
- Audit every `patchRun` caller and remove patch calls that only mirror runtime state.

Expected impact:

- reduce `workflowRuns.patchRun`
- reduce `dispatchRun`
- reduce stop/start/update spikes under reminder and redispatch loops

### Phase 3: Add missing visibility before structural optimization

Before changing storage model, add traces for still-missing hotspots:

- `GET /api/workflow-runs`
- `GET /api/workflow-runs/:run_id`
- `listWorkflowRuns`
- `getWorkflowRun`
- `writeWorkflowRunTaskRuntimeState`
- `appendWorkflowRunEvent`
- `listWorkflowRunEvents`
- `GET /api/workflow-runs/:run_id/task-tree`
- `GET /api/workflow-runs/:run_id/status`
- `GET /api/workflow-runs/:run_id/sessions`
- `GET /api/workflow-runs/:run_id/agent-io/timeline`
- `POST /api/workflow-runs/:run_id/agent-chat`

These routes are intentionally listed as gaps because the current implementation only traces a subset of workflow route handlers:

- `POST /api/workflow-runs`
- `POST /api/workflow-runs/:run_id/sessions`
- `GET /api/workflow-runs/:run_id/task-runtime`
- `GET /api/workflow-runs/:run_id/task-tree-runtime`
- `PATCH /api/workflow-runs/:run_id/orchestrator/settings`
- `POST /api/workflow-runs/:run_id/orchestrator/dispatch`

This closes the remaining blind spots without changing public behavior.

### Phase 4: Evaluate caching or sharding only after Phase 1 and 2

Do not jump straight to a storage rewrite. First verify the effect of removing duplicate reads and unnecessary registry writes.

If hotspots remain after that:

- add run-local in-memory cache with write invalidation
- consider splitting session/runtime/event stores by concern
- consider snapshot caching for task tree views

## Current Recommendation

The next optimization iteration should start with:

1. remove unnecessary `patchRun` calls from hot workflow runtime paths
2. stop full run hydration on every `task-runtime` and `task-tree-runtime` request
3. reuse already-loaded runtime/session state within the same request

These three changes are the highest-confidence path to bring down slow API warnings without changing the external API contract.
